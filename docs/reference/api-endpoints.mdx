## Authentication & tenant isolation

Every dashboard API route requires a signed-in session, and every query it runs
is scoped to the caller's organization:

- **Session routes** (`/api/tickets/*`, `/api/github/repos/*`, `/api/kb/*`,
  `/api/push/subscribe`, and the rest of the dashboard surface) resolve the
  org from the session. Requests without a session receive `401`; resources
  belonging to another org return `404` — object IDs are never enough to read
  or mutate data across orgs.
- **Bot ingest** (`POST /api/ingest`, Telegram/Email webhooks) authenticates
  with a per-org `bot_secret` bearer token; the org is resolved from that
  secret.
- **Widget chat** (`POST /api/widget/chat`) is public but authenticates by
  per-org widget token; all knowledge-base and prior-answer lookups are scoped
  to the org that owns the token.
- **GitHub webhook** (`POST /api/github/webhook`) verifies the HMAC-SHA256
  signature and resolves the org from the configured repository row.
- **MCP server** (`POST /api/mcp`) is public but authenticates every method
  except `initialize`/`notifications/initialized` via `Authorization: Bearer
  <key>`; the org is resolved from the API key. See [MCP
  Server](/integrations/mcp) for the tool surface. `GET /api/api-keys` and the
  create/revoke actions behind it are session routes like the rest of the
  dashboard.
- **Agent API** (`GET/POST /api/agent/*`) is the REST counterpart to the MCP
  server — same `Authorization: Bearer <key>` auth, same api_keys table, same
  org resolution, independent rate-limit quota. See [Agent
  API](/integrations/agent-api) for the endpoint list and OpenAPI spec.
